Core Concepts

Authentication & Modes

Machine clients use scoped API keys; dashboard users use JWT access + refresh tokens. Every credential carries a test or live mode that isolates its data. All credentials travel in the same header: 

Authorization: Bearer <api key or access token>

API keys

Format: <sk|pk>_<test|live>_<base58 random> — e.g. sk_test_4fzN…. The prefix tells you everything at a glance:

PrefixTypeUse
sk_test_ / sk_live_secretServer-side. Full API access up to its scope. Never expose in a browser or mobile app.
pk_test_ / pk_live_publishableClient-side identifiers (e.g. pairing with processor tokenization). Give them minimal scope.
  • Only an HMAC-SHA256 hash is stored — the plaintext is shown once at creation and can never be retrieved again.
  • Keys are managed via the API Keys endpoints (JWT required).
  • Revocation takes effect immediately (the server's auth cache is evicted on revoke).
  • An unknown, revoked, or expired key gets the same opaque 401 invalid_api_key response.

Scopes

Scopes are hierarchical: admin ⊇ write ⊇ read. A key holding admin passes any scope check. Insufficient scope returns 403 insufficient_scope.

Scope applies to the whole group

The minimum scope guards every method in the group, including GETs — e.g. listing charges requires a write-scoped key. Dashboard JWTs carry admin scope over the merchant's own account, which is why a JWT also works on the Customers endpoints.

Dashboard JWTs

Humans sign in once and hold two tokens: a short-lived access token (default 15 minutes) sent on every request, and a long-lived refresh token (default 30 days) used only to mint the next pair. Refresh tokens are single-use — each refresh rotates the token, and presenting an already-used one is rejected (a theft signal).

Refresh (rotate)

POST/v1/auth/refresh
no auth header — the body carries the credential
FieldTypeNotes
refresh_tokenstringrequiredThe current refresh token.

Returns 200 with a brand-new access + refresh pair. The old refresh token is revoked atomically in the same step. Reusing a rotated token returns 401 — if your client sees this unexpectedly, treat the session as compromised and re-login.

Logout

POST/v1/auth/logout
JWT
FieldTypeNotes
refresh_tokenstringrequiredThe refresh token to revoke.

Returns 204 always — logout is idempotent, even for unknown or already-revoked tokens. The short-lived access token simply expires on its own.

Token expiry in practice

  • An expired access token returns 401 with code token_expired — your cue to call /v1/auth/refresh and retry.
  • Any other invalid token returns 401 authentication_required.

Test vs live mode

Every API key and JWT carries a mode. Mode-bearing data — charges, plans, subscriptions, payouts — is filtered by it on every query:

  • A test-mode credential cannot read or touch live data (you get 404, as if it didn't exist) and vice versa.
  • Processor credentials are selected per request: a test request uses Stripe test keys / Plaid sandbox, a live request uses live keys. They never cross.
  • Test mode has a lower rate limit (default 100 req/min vs 1000 for live; see rate limits).
  • Customers and bank accounts are merchant-scoped but not mode-scoped — the same customer record is visible from both modes.
Develop in test, switch by swapping the key

Build your integration with an sk_test_… key, then deploy with an sk_live_… key. No code change is needed — the credential decides everything.

Credential hygiene

Secret keys and tokens are never logged by the service, and authentication failures are answered with deliberately vague messages. On your side: keep sk_ keys out of client code and version control, scope keys minimally (e.g. read for analytics jobs), and revoke anything you suspect leaked — revocation is immediate.

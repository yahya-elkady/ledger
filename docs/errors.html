Errors & Conventions

One error envelope, one pagination scheme, one money format — everywhere. Learn these once and every endpoint behaves the same way.

The error envelope

Every error — from any endpoint, any middleware, even unmatched routes and panics — uses this JSON shape:

{
  "error":      "validation_error",            // machine-readable code (table below)
  "message":    "amount must be a positive integer (minor units)",
  "param":      "amount",                      // only on validation errors: the offending field
  "request_id": "b3f1c2…"                      // echo this when reporting issues — it links to server logs
}

Error codes

CodeHTTPMeaning / what to do

idempotency_key_required400An authenticated POST was sent without an Idempotency-Key header.

invalid_api_key401Key is unknown, revoked, or expired (deliberately indistinguishable). Mint a new key.
token_expired401Access token past its TTL. Call POST /v1/auth/refresh and retry.
insufficient_scope403The key authenticated but lacks the route's minimum scope (scope table).
not_found404No such resource for your merchant and mode — cross-mode and cross-merchant access also looks like 404.
method_not_allowed405The path exists but not for this HTTP method.
idempotency_conflict409This idempotency key was already used for a different, still-in-flight or stored write. Also: 409 validation_error on duplicate registration email.

Idempotency

Every authenticated POST (create charge, create customer, refund, cancel, …) requires an Idempotency-Key header. Auth endpoints (/v1/auth/*) and inbound webhooks are exempt.

  • Use a unique random string per logical operation — UUIDv4 is perfect.
  • The first request executes; its response is cached for 24 hours.
  • Any retry with the same key (same merchant) replays the cached status + body verbatim and adds the header Idempotency-Replayed: true.
  • Server errors (5xx) are not cached — you can retry those for a fresh attempt.
  • A database uniqueness backstop on charges/payouts returns 409 idempotency_conflict if a duplicate ever races past the cache.
POST /v1/charges
Idempotency-Key: 0c5b2a4e-9f3d-4e5a-8c7b-1a2b3c4d5e6f
…network drops, you never saw the response…

POST /v1/charges                      ← identical retry, same key
Idempotency-Key: 0c5b2a4e-9f3d-4e5a-8c7b-1a2b3c4d5e6f
→ 201, Idempotency-Replayed: true     ← the customer was charged exactly once

Rate limiting

Limits are per identity over a sliding 60-second window:

IdentityDefault limitBucket
Live-mode API key1000 req/minper key
Test-mode API key100 req/minper key
Dashboard (JWT)300 req/minper merchant
Register / login10 req/minper IP

(Operators can tune these — see environment variables.) Every response on a limited route carries:

X-RateLimit-Limit:     100        # your limit for this window
X-RateLimit-Remaining: 97         # requests left
X-RateLimit-Reset:     1760000000 # unix seconds when the window resets

Exceeding the limit returns 429 with a Retry-After header and:

{ "error": "rate_limit_exceeded", "message": "rate limit exceeded", "retry_after": 12, "request_id": "…" }

Honor retry_after with backoff rather than hammering — repeated 429s are logged as an abuse signal.

Pagination

List endpoints return a collection envelope:

{ "data": [ … ], "next_cursor": "opaque-token" }
Query paramTypeNotes
limitintPage size. Default 20, max 100 (silently clamped).
cursorstringThe previous response's next_cursor. Opaque — don't construct or decode it. A garbled cursor returns 400 validation_error.

When next_cursor is absent or empty, you've reached the end. Cursor pagination applies to customers, charges, subscriptions, payouts, and dashboard transactions; plans and bank accounts return the full list. Charges and subscriptions also accept a ?status= filter.

Money & currencies

Amounts are always integers in the currency's minor unit. No decimals, no floats, ever. The API validates every amount + currency pair and never performs FX conversion.

Other conventions

  • Timestamps are RFC 3339 / ISO 8601 UTC strings ("2026-06-11T12:00:00Z"); request fields like trial_end use the same format.
  • Metadata fields accept any JSON object (≈ key-value tags) and are stored and returned verbatim. Don't put secrets or card data in them.
  • Deletes are soft — DELETE returns 204 and the resource stops appearing, but financial history is preserved internally.
  • CORS: browsers may call the API only from origins on the operator's allowlist (ALLOWED_ORIGINS); allowed request headers are Authorization, Content-Type, Idempotency-Key.
  • Security headers: every response carries X-Content-Type-Options: nosniff and X-Frame-Options: DENY.
  • Request bodies are capped at 1 MiB.

Deployment & Setup

For developers running the service: how to host it, every .env variable explained, what to check before production — and ideas for what to build on top.

The stack

One Go binary plus two backing services:

ComponentUsed forVersion
PostgreSQLAll persistent data (merchants, charges, ledger, audit log)16
RedisRate limiting, idempotency cache, API-key auth cache — safe to lose on restart7
GoBuilding/running the server (skip if using Docker)1.26

Stripe and Plaid accounts are optional in development — the server boots without their keys; only the routes that reach the processor need them.

Option A — Docker Compose (recommended)

  1. Configure

    cp .env.example .env
# then edit .env: set the four *_SECRET values (each ≥ 32 chars, and the two JWT
# secrets must differ). Generate strong values, e.g.:
#   openssl rand -base64 48

  2. Launch

    docker compose up --build

    This starts four services in order: postgres (provisions least-privilege DB roles on first init) → migrate (one-shot job that applies migrations/*.sql as the DDL-owning payments_migrations role, tracked in schema_migrations so re-runs are no-ops) → redis → api (only after migrations finish).

  3. Verify

    curl http://localhost:8080/health
# → {"status":"ok","db":"ok","redis":"ok"}

    Tear down with docker compose down (add -v to wipe the database volume).

The API image is a multi-stage build producing a static binary in a distroless, non-root image (no shell, no package manager). The app connects to Postgres as payments_app, a role with no DDL rights — schema changes can only happen through the migration job.

Option B — local toolchain

With Postgres and Redis already running locally:

cp .env.example .env          # set DATABASE_URL, REDIS_URL, and the secrets
make migrate-up               # apply migrations (golang-migrate CLI)
make run                      # start the server on :8080
make seed-test                # optional: test merchant + API key + sample data

Environment reference (.env)

Configuration is 100% environment-driven (a local .env is read in development; production should inject real environment variables). The process fails fast at startup if a required value is missing or a secret is too short — it never boots half-configured.

Server

VariableDefaultNotes
PORT8080HTTP listen port.
ENVdevelopmentdevelopment | production. Production enables extra safety checks (see below).
LOG_LEVELinfodebug | info | warn | error. Structured JSON logs (zerolog), one line per request.

Database & Redis

VariableDefaultNotes
DATABASE_URLrequiredPostgres DSN. Connect as the least-privilege payments_app role, never a superuser. sslmode=require in production — the server refuses to start with sslmode=disable when ENV=production.
DATABASE_MAX_CONNS25Connection pool ceiling.
DATABASE_MIN_CONNS5Pool floor (kept warm).
REDIS_URLredis://localhost:6379Rate limits + idempotency + auth cache. If Redis goes down, rate limiting fails open (requests pass) rather than taking the API down.

Secrets all required, ≥ 32 chars

VariableNotes
JWT_ACCESS_SECRETSigns dashboard access tokens.
JWT_REFRESH_SECRETSigns refresh tokens. Must differ from the access secret — startup fails if equal.
API_KEY_HMAC_SECRETHMAC-SHA256 secret for hashing API keys at rest. Rotating it invalidates every existing key — plan key re-issuance.
WEBHOOK_SIGNING_SECRETMaster secret from which per-endpoint webhook secrets are derived. Rotating it invalidates all endpoint secrets.

Production checklist

  • TLS at a reverse proxy (nginx/Caddy/ALB) — the app deliberately speaks plain HTTP internally. Set TRUST_PROXY_HEADERS=true once behind it.
  • Restrict /metrics at the proxy or network layer — it's unauthenticated by design (Prometheus scrape) and must not be public.
  • ENV=production — enforces TLS-to-Postgres and disables dev conveniences.
  • Secrets from a manager (Vault, AWS/GCP secrets, …) injected as env vars. Never commit .env — it's gitignored for a reason.
  • Run migrations as a deploy step (the compose migrate job pattern), not from the app — the app role can't run DDL.
  • Point processor webhooks at /v1/webhooks/stripe and /v1/webhooks/plaid and set the verification secrets — without them, async settlement (ACH charges, payouts) never updates.
  • Set ALLOWED_ORIGINS to exactly your dashboard origin(s).
  • Scrape /metrics and ship logs. Every request logs one JSON line carrying request_id (the same id returned in error envelopes); metrics cover request latency, charge volume, rate-limit hits, and webhook delivery outcomes.

What can you build on this API?

🖥 Merchant dashboard
A React/Vue SPA: register & login (JWT), revenue charts from /dashboard/overview, transaction feed, API-key management, customer CRUD. Everything it needs accepts JWTs — no secret keys in the browser.
🛒 E-commerce checkout
Your store backend tokenizes the card client-side (Stripe Elements), then calls POST /charges with a write key. Refund returns from the same API; webhooks reconcile orders.
📦 SaaS billing portal
Plans + subscriptions with trials: signup creates a customer and a subscription; subscription.updated webhooks drive dunning/feature flags; cancel-at-period-end for downgrades.
🤝 Marketplace / gig payouts
Charge buyers, then pay sellers: bank accounts + payouts with an admin key, with payout.paid/payout.failed webhooks feeding a seller-facing status page.
🎗 Donation platform
One-time gifts (charges, 12 currencies) and recurring giving (plans + subscriptions); test mode makes demo days safe.
📈 Ops & reconciliation tools
Slack alerts on charge.failed webhooks, a Grafana board on /metrics, nightly exports via list endpoints with a read key — plus the append-only audit trail underneath.

A typical full app combines three credentials: a JWT for the human dashboard, a write key for the money-in path, and an admin key locked away for payouts — with a webhook receiver keeping your database in sync.

Day-to-day development

  • make test — the unit suite runs without Docker or network (store fakes + in-memory Redis).
  • make test-integration — spins up real Postgres + Redis containers and runs the full request path end-to-end (register → key → charge → replay → 429 → mode isolation).
  • SQL changes: edit internal/store/queries/*.sql, run make generate (sqlc), add a numbered file in migrations/.
  • The double-entry ledger demo: go run ./cmd/smoke writes a balanced entry set against Postgres.

API Keys

Mint, inspect, and revoke the machine credentials your servers use. Key management is a dashboard concern, so these endpoints require a JWT — an API key cannot mint other API keys.

JWT only

The API key object

FieldTypeNotes
idstringUnique identifier (used to revoke).
namestringYour label, e.g. "backend server".
key_prefixstringFirst 12 characters of the plaintext (e.g. sk_test_4fzN) — enough to recognize a key in a list.
typestringsecret or publishable.
modestringtest or live.
scopestring[]Subset of read / write / admin (hierarchical — see scopes).
last_used_attimestampPresent once the key has authenticated.
expires_attimestampPresent if the key has an expiry; expired keys stop authenticating.
created_attimestampRFC 3339 UTC.

The full plaintext key appears in exactly one response: the create call, as the extra field key.

Create an API key

POST/v1/apikeys
Idempotency-Key required
FieldTypeNotes
namestringrequiredA label you'll recognize later.
typestringrequiredsecret (server-side) or publishable (client-side).
modestringrequiredtest or live.
scopestring[]requiredAt least one of read, write, admin.

Returns 201 with the key object plus the one-time plaintext:

{
  "id": "c91d…",
  "name": "backend server",
  "key_prefix": "sk_test_4fzN",
  "type": "secret",
  "mode": "test",
  "scope": ["write"],
  "created_at": "2026-06-11T12:00:00Z",
  "key": "sk_test_4fzN8q…"   ← shown once; only its HMAC hash is stored
}
curl -X POST http://localhost:8080/v1/apikeys \
  -H "Authorization: Bearer $ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -H "Idempotency-Key: $(uuidgen)" \
  -d '{"name": "checkout backend", "type": "secret", "mode": "test", "scope": ["write"]}'

List API keys

GET/v1/apikeys

Returns 200 with {"data": [key, …]} — metadata only (prefix, never the key), active keys only.

Retrieve an API key

GET/v1/apikeys/{id}

Returns 200 with the key's metadata, or 404 not_found.

Revoke an API key

DELETE/v1/apikeys/{id}

Revocation is immediate — the server also evicts its auth cache, so the key stops authenticating on the very next request, not after a cache TTL.

Returns 204, or 404 not_found.

Key strategy

Mint one key per system with the smallest scope it needs: read for reporting jobs, write for the checkout path, admin only for the service that initiates payouts. Rotate by minting the replacement first, deploying it, then revoking the old key — revocation is instant, so do it in that order.

Dashboard

Read-only aggregates for building a merchant dashboard UI: headline numbers and a unified transaction feed. JWT-only, and scoped to the mode carried by the token.

JWT only mode-isolated

Overview

GET/v1/dashboard/overview

Returns 200 with current aggregates for your merchant in the token's mode:

FieldTypeNotes
modestringThe mode these numbers cover.
charge_countintAll charges, any status.
charge_succeededintCount of succeeded charges.
charge_failedintCount of failed charges.
gross_volumeintSum of succeeded charge amounts, minor units.
active_subscriptionsintCurrently active subscriptions.
pending_payoutsintPayouts not yet settled.
recent_failed_chargescharge[]Up to 10 most recent failed charges, for an "attention needed" panel.
 
curl http://localhost:8080/v1/dashboard/overview \
  -H "Authorization: Bearer $ACCESS_TOKEN"
{
  "mode": "test",
  "charge_count": 128,
  "charge_succeeded": 119,
  "charge_failed": 9,
  "gross_volume": 482350,
  "active_subscriptions": 23,
  "pending_payouts": 2,
  "recent_failed_charges": [ { "id": "…", "status": "failed", "failure_code": "card_declined", … } ]
}

Transactions

GET/v1/dashboard/transactions

A paginated transaction feed (currently surfaces charges — the primary transaction type). Accepts the same filters as List charges:

Query paramNotes
statusoptionale.g. succeeded, failed, refunded.
limitoptionalDefault 20, max 100.
cursoroptionalFrom the previous next_cursor.

Returns 200 with {"data": [charge, …], "next_cursor": "…"}.

Building a dashboard?

Pair these endpoints with Customers (JWTs work there too) and API Keys for a complete self-serve merchant portal — register, login, mint keys, watch the money. Dashboard JWT traffic is limited to 300 requests/min per merchant by default, comfortable for a SPA.

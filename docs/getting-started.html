Getting Started

From zero to a test charge in five requests. You need a running instance (see Deployment & Setup — docker compose up --build is the quickest path) and curl or any HTTP client.

Check it's alive first

curl http://localhost:8080/health should return {"status":"ok","db":"ok","redis":"ok"}.

  1. Register a merchant account

    New merchants start in test mode. The response contains your dashboard JWT tokens and merchant record.

    curl -X POST http://localhost:8080/v1/auth/register \
  -H "Content-Type: application/json" \
  -d '{
    "email": "you@example.com",
    "password": "a-strong-password",
    "business_name": "Acme Store"
  }'

    Response · 201

    {
  "access_token":  "eyJhbGciOi...",
  "refresh_token": "eyJhbGciOi...",
  "token_type":    "Bearer",
  "expires_in":    900,
  "merchant": { "id": "…", "email": "you@example.com", "business_name": "Acme Store", "mode": "test", ... }
}

    Password must be at least 8 characters. The access token lasts 15 minutes — use POST /v1/auth/refresh to rotate (see Authentication).

  2. Create an API key

    API keys are managed with your JWT, and creating one is a POST, so it needs an Idempotency-Key header (any unique string; a UUID is ideal).

    curl -X POST http://localhost:8080/v1/apikeys \
  -H "Authorization: Bearer $ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -H "Idempotency-Key: $(uuidgen)" \
  -d '{
    "name": "backend server",
    "type": "secret",
    "mode": "test",
    "scope": ["admin"]
  }'

    Response · 201

    {
  "id": "…",
  "name": "backend server",
  "key_prefix": "sk_test_4fzN",
  "type": "secret",
  "mode": "test",
  "scope": ["admin"],
  "created_at": "2026-06-11T12:00:00Z",
  "key": "sk_test_4fzN…"        ← shown ONCE — store it now, it is never retrievable again
}
    Save the key

    Only an HMAC hash is stored server-side. If you lose the plaintext, revoke the key and mint a new one.

  3. Create a customer

    curl -X POST http://localhost:8080/v1/customers \
  -H "Authorization: Bearer sk_test_..." \
  -H "Content-Type: application/json" \
  -H "Idempotency-Key: $(uuidgen)" \
  -d '{
    "email": "jane@example.com",
    "name": "Jane Doe",
    "metadata": {"internal_ref": "cust-42"}
  }'

    Returns 201 with the customer object; keep the id for the charge.

  4. Take a test charge

    Amounts are integer minor units — 2500 is $25.00. The payment method id is a processor token (e.g. Stripe's test token pm_card_visa); raw card numbers never touch this API.

    curl -X POST http://localhost:8080/v1/charges \
  -H "Authorization: Bearer sk_test_..." \
  -H "Content-Type: application/json" \
  -H "Idempotency-Key: $(uuidgen)" \
  -d '{
    "amount": 2500,
    "currency": "USD",
    "customer_id": "<customer id>",
    "payment_method_id": "pm_card_visa",
    "processor": "stripe",
    "description": "First test charge"
  }'

    Response · 201

    {
  "id": "…",
  "amount": 2500,
  "currency": "USD",
  "status": "succeeded",
  "processor": "stripe",
  "mode": "test",
  "refunded_amount": 0,
  ...
}

    A declined card also returns 201 — as a recorded charge with "status": "failed" plus failure_code / failure_message. Retrying with the same Idempotency-Key replays this exact response instead of charging again.

  5. See the result

    List charges with your API key, or pull dashboard aggregates with your JWT:

    curl http://localhost:8080/v1/charges?limit=10 \
  -H "Authorization: Bearer sk_test_..."

curl http://localhost:8080/v1/dashboard/overview \
  -H "Authorization: Bearer $ACCESS_TOKEN"

Where to go next

  • Plans + Subscriptions — recurring billing.
  • Bank Accounts + Payouts — move money out (requires an admin-scope key).
  • Webhooks — get notified when charges settle, payouts pay, subscriptions cancel.
  • Errors & Conventions — the error envelope, pagination, rate limits.
Seed fixtures instead

Running locally without Docker? make seed-test creates a test merchant, an API key (plaintext printed once to the console), a customer, and sample charges in one go.

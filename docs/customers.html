Customers

A customer is one of your end-users — the person you charge or subscribe. Customers are scoped to your merchant account (visible from both test and live mode) and support free-form metadata.

API key or JWT write scope

Create a customer

POST/v1/customers
Idempotency-Key required
FieldTypeNotes
emailstringoptional
namestringoptional
metadataobjectoptionalAny JSON object.

Returns 201 with the customer object.

curl -X POST http://localhost:8080/v1/customers \
  -H "Authorization: Bearer sk_test_..." \
  -H "Content-Type: application/json" \
  -H "Idempotency-Key: $(uuidgen)" \
  -d '{"email": "jane@example.com", "name": "Jane Doe"}'

List customers

GET/v1/customers
Query paramNotes
limitoptionalDefault 20, max 100.
cursoroptionalFrom the previous next_cursor.

Returns 200 with {"data": [customer, …], "next_cursor": "…"}, newest first.

Retrieve a customer

GET/v1/customers/{id}

Returns 200 with the customer, or 404 not_found if the id doesn't exist for your merchant (or was deleted).

Update a customer

PUT/v1/customers/{id}

Takes the same body as create and replaces the mutable fields (email, name, metadata) — omitted fields are cleared, so send the full desired state.

Returns 200 with the updated customer, or 404.

curl -X PUT http://localhost:8080/v1/customers/1f3c… \
  -H "Authorization: Bearer sk_test_..." \
  -H "Content-Type: application/json" \
  -d '{"email": "jane+new@example.com", "name": "Jane Doe", "metadata": {"tier": "pro"}}'

Delete a customer

DELETE/v1/customers/{id}

Soft-deletes the customer — it disappears from reads and lists, but historical charges that reference it are unaffected.

Returns 204 on success, 404 if not found.

Two credentials work here

Customers are the one resource group that accepts either a write-scoped API key or a dashboard JWT — so both your backend and your dashboard UI can manage them. Every mutation is recorded in the audit log either way.

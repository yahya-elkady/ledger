Webhooks

When a charge settles, a payout pays, or a subscription changes, the service POSTs a signed event to your registered endpoints — with retries, so a blip on your side doesn't lose the event. This page covers the events you receive, how to verify them, and the two inbound endpoints the processors call.

Event types

The delivery

Each event is an HTTP POST to your endpoint:

POST https://your-app.example.com/payments/webhook
Content-Type: application/json
X-Payments-Timestamp: 1760000000
X-Payments-Signature: sha256=7f83b1657ff1fc53b92dc18148a1d65dfc2d4b1fa3d677284addd200126d9069

{
  "id":      "d41e…",            // delivery id — use it for de-duplication
  "type":    "charge.succeeded",
  "created": 1760000000,          // unix seconds
  "data":    { …the charge/payout/subscription object… }
}
  • Respond with any 2xx quickly (within 30 seconds) — do your processing async if it's slow.
  • Non-2xx or a timeout schedules a retry with exponential backoff: 60s × 2^attempt (≈ 2 min, 4 min, 8 min, 16 min by default), up to 5 total attempts, then the delivery is dead-lettered.
  • Retries mean you can see the same event more than once — de-duplicate on id.
  • Events can also arrive out of order — trust the resource's status over arrival order.

Verifying signatures

Every delivery is signed with your endpoint's secret (shown once when the endpoint is registered). The signature is:

X-Payments-Signature: sha256=hex( HMAC-SHA256( secret, "<timestamp>.<raw body>" ) )

Verify before trusting anything: recompute over the raw request bytes (not re-encoded JSON), compare in constant time, and reject stale timestamps (the service tolerates ±300 seconds) so captured requests can't be replayed.

Python

import hashlib, hmac, time

def verify(raw_body: bytes, ts: str, sig: str, secret: str) -> bool:
    if abs(time.time() - int(ts)) > 300:               # replay window
        return False
    mac = hmac.new(secret.encode(), f"{ts}.".encode() + raw_body, hashlib.sha256)
    expected = "sha256=" + mac.hexdigest()
    return hmac.compare_digest(expected, sig)           # constant-time

Node.js (Express — note the raw body)

const crypto = require("crypto");

app.post("/payments/webhook",
  express.raw({ type: "application/json" }),           // keep the RAW bytes
  (req, res) => {
    const ts  = req.get("X-Payments-Timestamp");
    const sig = req.get("X-Payments-Signature");
    if (Math.abs(Date.now() / 1000 - Number(ts)) > 300) return res.sendStatus(400);

    const expected = "sha256=" + crypto
      .createHmac("sha256", process.env.ENDPOINT_SECRET)
      .update(`${ts}.`).update(req.body)               // req.body is a Buffer here
      .digest("hex");

    const ok = expected.length === sig.length &&
      crypto.timingSafeEqual(Buffer.from(expected), Buffer.from(sig));
    if (!ok) return res.sendStatus(400);

    const event = JSON.parse(req.body);
    res.sendStatus(200);                               // ack fast, process async
    handleEvent(event);
  });
The #1 verification bug

Parsing the JSON and re-serializing it before hashing. Key order or whitespace changes and the HMAC never matches. Always hash the exact raw bytes you received.

Endpoint secrets

Each endpoint gets its own secret, derived from the service's master signing secret — it is shown once at registration and never stored server-side, and one endpoint's secret reveals nothing about another's.

Endpoint registration API: not yet exposed

The delivery pipeline (signing, retries, dead-lettering) is fully operational, but the self-serve /v1/webhook-endpoints CRUD is still on the roadmap — endpoints are currently registered by the operator directly in the database (webhook_endpoints table). Ask your operator to register your URL and event subscriptions, and hand you the endpoint secret.

Inbound processor webhooks (operator concern)

These two endpoints receive events from the processors — they're what advances pending charges, payouts, and subscription statuses. Point Stripe and Plaid at them when deploying; application clients never call them.

POST/v1/webhooks/stripe

Verified against the Stripe-Signature header using STRIPE_WEBHOOK_SECRET (from the Stripe dashboard's endpoint settings).

POST/v1/webhooks/plaid

Verified against the Plaid-Verification header (Plaid's JWT-based scheme).

  • Invalid or missing signature → 400, and the event is never processed.
  • Valid events the service doesn't care about are acknowledged with 200 and ignored (so processors don't retry forever).
  • Handled events update the matching charge / payout / subscription, write an audit row, and are relayed to your merchant webhook endpoints as the events above.
  • These routes are deliberately not identity-rate-limited — processors burst.

Introduction

Ledger API

A Stripe-like payments REST API. It takes card charges and ACH bank transfers, runs subscriptions, pays out to bank accounts, and notifies you of every state change via signed webhooks — behind one consistent, multi-tenant JSON interface that wraps Stripe and Plaid.

REST · JSON test / live modes API keys + JWT Idempotent writes Signed webhooks

Base URL

http://localhost:8080/v1        # local development (default port 8080)
https://api.example.com/v1      # production, TLS terminated at your proxy

All requests and responses are JSON (Content-Type: application/json). Request bodies are capped at 1 MiB, and unknown JSON fields are rejected with a validation_error — typos never pass silently.

Two ways to authenticate

  • API keys — for server-to-server integrations. Send Authorization: Bearer sk_test_… (or sk_live_…). Keys carry a scope (read < write < admin) and a mode (test / live).
  • JWT bearer tokens — for dashboard users (humans). Obtained from POST /v1/auth/register or /v1/auth/login; short-lived access tokens with rotating refresh tokens.

See Authentication & Modes for the full rules.

Five things to know

  1. Amounts are integer minor units. 1099 means $10.99 (USD) — and 1099 means ¥1099 for zero-decimal JPY. Floats are never used. 12 ISO 4217 currencies are supported (list).
  2. Test and live mode never mix. Every credential carries a mode; a test key cannot see live data (it gets 404), and live processor credentials are never used for a test request.
  3. Every authenticated POST needs an Idempotency-Key header. Retries replay the original response instead of charging twice (details).
  4. Responses are rate-limited per identity with X-RateLimit-* headers on every response and 429 + Retry-After when exceeded.
  5. State changes push webhooks. Charges, payouts, and subscriptions emit signed events to your registered endpoints, with retries (verification guide).

PCI scope

This API never stores card numbers, CVVs, or bank credentials. Charges and subscriptions reference processor-tokenized payment methods (e.g. Stripe pm_… ids produced by client-side tokenization such as Stripe Elements), keeping your integration in SAQ-A territory.
